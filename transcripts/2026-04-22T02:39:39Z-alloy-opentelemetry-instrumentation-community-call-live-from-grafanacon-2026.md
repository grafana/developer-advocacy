## Introduction to O Tel and Alloy

For those new to O Tel and Alloy, the O Tel collector and Alloy are open-source collectors that collect, process, and export telemetry data. Many people are drawn to these collectors because of their flexibility. You can store the data collected with these collectors in the backend of your choice and change them as needs arise.

### Starting with Alloy

Now, let's start with Alloy. Bijal, there are currently **525,000 instances of Alloy**. When it launched two years ago, was that the expectation?

> Gosh, it’s kind of hard for me to say because I wasn’t actually on the Alloy team back then, but definitely, I think it’s a sign that it has fulfilled a strong demand. Whether it was the expectation to scale this far, I’m not sure, but it’s really cool that it has. It shows a lot about what people need.

We were also chatting about how there are almost **8 million weekly Docker pulls** for Alloy. For context, the O Tel collector combined has **4.9 million pulls**. What do you think is driving this adoption?

> I think it’s just a tried and tested collector solution. It encompasses so many different ecosystems and formats. If you need a collector solution, even for niche setups, Alloy usually has components for you. It suits a wide variety of situations and scenarios, making it very appealing.

### Role of Community Feedback

Our community plays an important role in shaping our product direction. How do you think community feedback has contributed to shaping Alloy?

> It’s been a back-and-forth the whole time. The community doesn’t shy away from addressing issues. We receive a lot of GitHub issues and engagement in the community Slack. We try to respond as much as we can. There have been instances where we proposed new features and received pushback from the community, making it bidirectional. Recently, we’ve also started designing in the open, so many of our design docs are now PRs that community members can comment on. This allows the community to have a say in how we implement certain features.

### Alloy’s OpenTelemetry Engine

I’m curious about the community's feedback on Alloy's open telemetry engine. Bijal, could you give us more background on what these engines are?

> Yes, Alloy now has two engines. The default engine is the underlying runtime that has been in Alloy since its inception. We recently introduced the new OpenTelemetry engine, which is essentially the upstream OpenTelemetry collector runtime embedded within Alloy. This allows for a native OTel experience when running Alloy.

Why do we have two engines instead of just replacing the existing one?

> The existing one is very popular, as indicated by the millions of Docker pulls. We don’t want to eliminate something that people like and need. A key feature of the default engine is that it spans across ecosystems and different data formats. While the observability world is moving toward standards, not everyone is there yet. We need a solution that takes that into account. The default engine also contains many OpenTelemetry components. 

If someone is using the vanilla OTel collector, what would the migration look like? Is it just as simple as copy and paste?

> Yes, it would be. You would take your existing configuration and pass it to the OTel engine, running it with the same CLI as upstream. The transition requires very little effort. Just check if the components you’re using are available in Alloy, as some distributions bundle different components. If they are available, it’s just a matter of taking your configuration, pointing it, and running it.

### Community Reactions and Questions

What has the community reaction been like? Is there ongoing conversation about shaping these engines?

> The hotel engine's development has really been driven by community reaction. There’s been a definition of a collective distribution that sparked a lot of work to introduce this new engine. 

What upstream components would people be excited about using natively with Alloy?

> Any components in the default engine are included in the OTel engine. We initially included components in the experimental release based on what’s in the default engine and any required for Grafana's Act 3 products. As we evolve, we’ll introduce more components based on community requests and our own developments.

### Questions from the Audience

We received a question from the audience: "We are already running on OTel Collector and sending data to LGTM. Should we move to Alloy or stay on OTel Collector?"

> It really depends on your needs and support requirements. A significant gain with the OTel engine is support. Alloy offers more features, but if your current instance is meeting your needs, you can definitely stick with it.

Can you tell us more about the Alloy engine extension?

> The Alloy engine extension allows you to run the default engine pipeline in parallel to the OTel engine pipeline. You can choose aspects of each without needing different deployments; it can all run in a single collector instance.

Is there a case where we should use both engines?

> A strong case for using both is during migration. If you're using Alloy configuration syntax and want to try out the OTel engine, you can either spin up a different deployment or keep it in a single deployment and port configuration between engines.

### Fleet Management Overview

Misha, let’s shift gears to fleet management. For those unfamiliar, how would you explain fleet management to a beginner?

> Fleet management is basically a control plane for a collector. It provides a bird's-eye view of all your collectors, allowing you to see how often data comes in and to update remote configurations.

I’ve heard that there are **373,000 collectors managed remotely**. What does that mean?

> It means those collectors are checking in on average once a minute, driven by large customers and some smaller ones. The protocol itself is open source, and some have implemented their own versions of fleet management using our defined protocol.

### Attribute-Based Pipeline Matching

Could you explain the concept of attribute-based pipeline matching?

> Sure! It's similar to alert manager matching logic. You define a set of attributes for a collector and specify which pipeline should be delivered to that collector based on those attributes. You can also use negations and regular expressions for more complex scenarios.

I heard there are on-demand pipelines that stay disabled until needed. What’s the cost story around that?

> This is useful during active incidents when you need granular data for a short time. You can avoid constant high resource consumption and costs by enabling those pipelines only when necessary.

### Future of Fleet Management

What does the roadmap look like for fleet management in the next six months?

> We have exciting front-end updates coming, including an overview of telemetry flow through collectors, cost analysis features, and visual updates to help configure Alloy instances. On the performance side, we’re also working on stability improvements.

### OpAmp and Grafana's Role

Ted, can you explain what OpAmp is?

> OpAmp stands for OpenTelemetry Control Plane Protocol. While Fleet Management provides a complete end-to-end solution, OpAmp is just a protocol that helps manage collectors, including upgrading them.

Grafana is actively contributing to the OpenTelemetry spec. What has that experience been like?

> It’s been interesting. OpenTelemetry requires careful consideration to avoid breaking changes across implementations. We gather feedback and prototype before finalizing standards, ensuring that all implementations are consistent across different languages.

### Installation and Rollout Challenges

What challenges do organizations face when rolling out OpenTelemetry?

> The main challenge is coordinating a rollout across multiple applications and teams. While installing OpenTelemetry in a single application isn't complicated, scaling that to a thousand applications can be daunting. We aim to provide tools that allow for system-wide deployment, making it easier for operations teams to manage.

### Instrumentation Hub and Beyla

Now, let’s talk about Instrumentation Hub and Beyla. How do they fit into the big picture?

> Instrumentation Hub and Beyla help achieve baseline service visibility by providing metrics and service graphs instantly. We plan to release the ability to do instrumentation with OpenTelemetry SDKs through Beyla soon.

Grafana donated Beyla to OpenTelemetry as OBI. What impact has that had?

> The impact has been better than expected. We’ve gained many developers working on the project, which has increased contributions significantly. We now have more maintainers outside of Grafana than within.

### Conclusion

Thank you all for the insightful discussion today! We have another AI community call at 6:00 PM, so stay tuned for that. 

Thank you, everyone!
